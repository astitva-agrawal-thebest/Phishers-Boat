# Folder Structure

## Detailed Component Breakdown

### 1. Extension Root (`ai-phishing-extension/`)
#### Manifest and Core Configuration
- **manifest.json**: Defines extension permissions, components, and behavior
  - Specifies Manifest V3 format
  - Declares background service worker
  - Sets content script injection rules
  - Defines action (toolbar button) and popup
  - Lists required permissions (activeTab, scripting, storage)
  - Specifies host permissions for backend communication
  - Lists web-accessible resources

#### Background Scripts (`background/`)
- **scanner.js**: Main orchestration thread
  - Implements the threat analysis pipeline
  - Coordinates local and cloud analysis
  - Manages caching and session storage
  - Handles message routing between components
  - Updates browser action (toolbar badge)
- **reputationClient.js**: Handles backend communication
  - Abstracts HTTP requests to localhost:3000
  - Manages API key retrieval from storage
  - Implements timeout and retry logic
  - Transforms backend responses to internal format
- **model_weights.json**: 
  - JSON file containing pre-trained model parameters
  - Format: { "feature_name": weight_value, ... }
  - Generated by train_model.py from dataset.csv
  - Used by analyzeWithLocalModel() for scoring

#### Content Scripts (`content/`)
- **injector.js**: 
  - Runs in context of web pages
  - Extracts textual content and links
  - Implements debouncing to limit processing frequency
  - Sends data to background worker via messaging
  - Receives and displays security banners
  - Handles SPA navigation via MutationObserver
- **injector.css**:
  - Styles for security notification banners
  - Animations for attention-grabbing effects
  - Responsive design for different viewport sizes
  - Accessibility-focused color schemes
  - Dark/light mode adaptability

#### Popup Interface (`popup/`)
- **popup.html**: 
  - Structure of extension to the page
  - Semantic HTML5 structure
  - Placeholders for dynamic content
  - ARIA labels for accessibility
- **popup.css**:
  - Custom styling matching extension theme
  - Flexbox/Grid layout for responsive design
  - Hover/active/focus states for interactive elements
  - Transition animations for smooth UX
  - Adaptive to system prefers-color-scheme
- **popup.js**:
  - State management for popup UI
  - Handles user interactions (buttons, inputs, lists)
  - Communicates with background via messaging
  - Renders dynamic content (statistics, lists, settings)
  - Implements manual URL and QR code scanners
  - Manages session trust/blocklist operations

#### Dataset (`dataset/`)
- **dataset.csv**:
  - Training data used to generate model_weights.json
  - Columns: URL, label (0=safe, 1=phishing), features...
  - Used during development/training phase
  - Not required for extension runtime (only weights needed)

#### Assets (`assets/`)
- **icon*.png**: Extension icons in various sizes
  - 16x16: Toolbar icon (default size)
  - 32x32: Context menu and some UI places
  - 48x32: Various browser UI locations
  - 128x128: Web store listing and detailed views

#### Utility Scripts
- **train_model.py**: Python script to train the local model
  - Reads dataset.csv
  - Extracts features and trains logistic regression or similar
  - Outputs model_weights.json
  - For research/development purposes
- **train_vanilla.py**: Baseline training script for comparison
- **get-pip.py**: Helper to install Python in restricted environments

### 2. Backend Server (`backend/`)
- **server.js**: Main application entry point
  - Initializes Express.js application
  - Configures middleware (cors, body-parser, dotenv)
  - Defines API routes (/api/url-reputation, /health)
  - Implements error handling and logging
  - Starts HTTP listener on specified port
- **package.json**: 
  - Project metadata and dependencies
  - Scripts: "start": "node server.js"
  - Dependencies: express, cors, dotenv
- **package-lock.json**: Exact dependency versions for reproducible builds
- **.env**: 
  - Environment variables (NOT committed to version control)
  - Contains: VT_API_KEY=your_actual_key_here
  - May contain: PORT=3000, HOST=0.0.0.0
  - Loaded by dotenv at runtime

### 3. Node Runtime (`node/`)
- Pre-bundled Node.js executable for convenience
- Ensures consistent runtime across development environments
- Includes:
  - node.exe: JavaScript runtime
  - npm: Package manager
  - npx: Package executor
  - Complete standard library
- Note: This is for developer convenience; production deployments should use platform-appropriate Node.js installation

## Key Design Principles Reflected in Structure

### Separation of Concerns
- **distinct directories** for background logic, content scripts, and UI
- **clear separation** between extension (client) and backend (server)
- **isolated configuration** via manifest.json and .env files
- **separate concerns** of data collection, analysis, communication, and presentation

### Modularity
- **background/** contains independent modules (scanning, reputation)
- **content/** focuses solely on page interaction and messaging
- **popup/** manages all user interface elements
- **dataset/** separates training data from runtime components
- **backend/** isolates server logic from extension code

### Scalability and Maintainability
- **flat hierarchy** where appropriate for easy navigation
- **consistent naming** conventions across files
- **separation of configuration** from code
- **minimal dependencies** in extension (no build process required)
- **clear entry points** (manifest.json, server.js)

### Security Boundaries
- **background/** cannot directly access page DOM
- **content/** cannot access chrome.* APIs except messaging
- **communication** strictly through validated message passing
- **backend** secrets stored in environment, not code
- **no direct** extension-to-third-party communication

### Development vs Production
- **source code** in clearly marked directories
- **build artifacts** would be separate (not committed)
- **documentation** segregated from source
- **utility scripts** kept with source but marked as dev-only
- **node/** directory optionally omitted in production deployments

## Typical Workflow Paths Through the Structure

### Development Cycle
1. Modify background/scanner.js → Test via chrome://extensions reload
2. Adjust content/injector.js → Reload extension and test on test pages
3. Update popup/popup.js → Open popup to see changes
4. Adjust backend/server.js → Restart node server and test API
5. Update model weights → Replace background/model_weights.json
6. Adjust styling → Edit respective .css files and refresh

### Production Build Process
1. Copy ai-phishing-extension/ directory as-is
2. Copy backend/ directory (excluding node_modules)
3. Create production build:
   - npm ci in backend/ (clean install from package-lock.json)
   - Optionally minify backend JavaScript
   - Verify manifest.json permissions are production-appropriate
4. Package extension:
   - Zip ai-phishing-extension/ contents for web store submission
   - Ensure no development files (like train_*.py) included
5. Deploy backend:
   - Set VT_API_KEY in hosting platform environment
   - Deploy node/ server.js with process manager (PM2, systemd, etc.)
   - Configure domain and SSL as needed
6. Update extension manifest:
   - Change background/reputationClient.js BACKEND_URL to production endpoint

## Special Directories Explanation

### Why Separate dataset/?
- Training data is large and only needed during model development
- Runtime only requires the much smaller model_weights.json
- Allows independent versioning of data vs. model
- Facilitates experimentation with different training sets

### Why Separate assets/?
- Icons are required by manifest.json at specific paths
- Keeping them organized prevents clutter in root directory
- Enables easy replacement for branding or updates
- Ensures all required sizes are present and accounted for

### Why Include node/?
- Eliminates environment setup barriers for new contributors
- Ensures consistent behavior across development machines
- Useful for training scripts that require Python dependencies
- Can be omitted in production deployments to reduce size
- Provides self-contained development experience

### Why Documentation Separate?
- Keeps source directories focused on implementation
- Allows documentation to evolve independently
- Facilitates contributions focused solely on documentation
- Enables different generation/publishing workflows
- Follows conventional project layout

## Navigation Guidelines

### For New Contributors
1. Start with manifest.json to understand extension capabilities
2. Review background/scanner.js for core logic flow
3. Examine content/injector.js for data collection mechanism
4. Study popup/popup.js for user interface implementation
5. Review backend/server.js for server-side logic
6. Check data-flow.md for end-to-end data movement
7. Refer to architecture.md for component relationships

### For Debugging Specific Issues
- **Popup/UI issues**: Examine popup/ directory and background messaging
- **Detection accuracy**: Review background/scanner.js and model_weights.json
- **Communication failures**: Check background/reputationClient.js and backend/server.js
- **UI styling problems**: Look at content/injector.css and popup/popup.css
- **Data storage issues**: Review background/scanner.js storage interactions
- **Performance concerns**: Check debouncing logic in content/injector.js

### For Extending Functionality
- **New detection engines**: Add to background/scanner.js and update pipeline
- **New reputation sources**: Modify background/reputationClient.js
- **New UI features**: Extend popup/ and/or content/injector.js
- **New storage types**: Update background/scanner.js storage interactions
- **New manifest permissions**: Edit manifest.json and update documentation
- **Backend endpoints**: Modify backend/server.js and related client code

## Platform-Specific Considerations

### Chromium-Based Browsers (Chrome, Edge, Brave)
- Primary development and testing target
- Uses standard WebExtensions API with Manifest V3
- All features implemented as specified
- Full compatibility expected

### Firefox
- Minor API differences handled via polyfills where needed
- Manifest V3 support with some variations
- May require slight adjustments to manifest permissions
- Core functionality identical

### Safari
- Requires Safari App Extension format (different structure)
- Not currently supported without significant adaptation
- Future work would involve adapting core logic to Safari APIs

## File Naming Conventions
- **snake_case**: Used for all files and directories
- **descriptive names**: Files clearly indicate their purpose
- **extension consistency**: 
  - .js for JavaScript files
  - .json for JSON data
  - .css for stylesheets
  - .html for HTML structures
  - .py for Python scripts
- **index files**: Not used; explicit naming preferred for clarity
- **test files**: Would follow *.test.js pattern if added

## Build and Deployment Notes
- **No build step required** for extension (interpreted JavaScript)
- **Backend** requires `npm install` followed by `node server.js`
- **Production builds** should:
  - Exclude development scripts (train_*.py, get-pip.py)
  - Ensure .env is not committed (add to .gitignore)
  - Use minified browser-appropriate JavaScript if needed
  - Set appropriate cache headers for static assets
- **Version bumping**:
  - Update manifest.json version field
  - Update package.json version if tracking backend versions
  - Follow semantic versioning guidelines
